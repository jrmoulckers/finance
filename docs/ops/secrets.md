# Secrets & Variables — Environment Parity

Last audited: 2026-06-10  
Authoritative source for which secrets each environment needs.

> This audit records names only. GitHub APIs used here do not expose secret values, and no secret values were printed or stored. `Set?` is based on GitHub repository/environment secret and variable names; deploy server `.env` contents were not inspected.

Legend: `✅ env secret/var` = set directly on that GitHub Environment; `✅ repo secret/var` = set at repository scope and inherited by jobs; `✅ built-in` = supplied by GitHub Actions; `❌ MISSING` = not visible from the audited GitHub APIs.

## Setup checklist for a new environment

1. Create the GitHub Environment under **Settings → Environments** and apply the required reviewer/branch rules for its risk level.
2. Add the environment-scoped deploy secrets used by its workflows: `DEPLOY_HOST`, `DEPLOY_SSH_KEY`, `DEPLOY_USER`, `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `POWERSYNC_URL`, `VERCEL_TOKEN`, `VERCEL_ORG_ID`, and `VERCEL_PROJECT_ID` as applicable.
3. Add `DEPLOY_PATH` as an environment variable when the target path is not `~/finance`.
4. Copy the matching deploy runtime template (`deploy/.env.example` for production-like hosts or `deploy/.env.staging.example` for staging), plus backup and monitoring templates if those services run in the environment.
5. Populate optional monitoring/beta values (`SENTRY_DSN`, `BETA_ALLOWED_EMAILS`, Sentry DSNs, monitor host/port values) or document why they are intentionally blank.
6. For release-capable environments, add platform signing and store API secrets for Android, iOS, and Windows before enabling release workflows.
7. Run the GitHub API name audit from this document and compare the new environment against staging/production before the first deploy.
